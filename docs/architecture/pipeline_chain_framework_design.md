# Pipeline Chain Framework Design

## 1. Purpose

This document defines a **generic pipeline chain execution model** for
batch processing systems.

The goal is to enable a framework that allows multiple processing stages
to be executed sequentially while keeping responsibilities clearly
separated between orchestration logic and processing logic.

Key objectives:

-   Execute **multi-stage processing with a single CLI command**
-   Execute the next stage **only if the previous stage succeeds**
-   Define **artifact contracts between stages**
-   Unify runtime context such as date, target_dir, max_images, and
    config
-   Persist pipeline execution results as summary data and JSON
    artifacts
-   Avoid inflating the responsibilities of the base processing
    framework

------------------------------------------------------------------------

## 2. Scope

### Included

-   CLI-based pipeline execution (--pipeline)
-   Sequential stage chain execution
-   Artifact contracts between stages
-   Runtime context propagation
-   Failure-stop rules
-   Pipeline execution summary output
-   Persistent pipeline summary JSON

### Excluded

-   Arbitrary stage ordering
-   Dynamic pipeline construction
-   Stage-specific resource limits
-   Framework-level redesign

------------------------------------------------------------------------

## 3. Terminology

### Stage

A processing unit that can be executed independently.

Examples may include:

-   data extraction
-   data transformation
-   evaluation
-   ranking

Each stage receives input artifacts and produces output artifacts.

------------------------------------------------------------------------

### Pipeline

A sequence of stages executed in a defined order.

Example conceptual flow:

Stage A ↓ Stage B ↓ Stage C

------------------------------------------------------------------------

### Artifact

A result generated by a stage that can be consumed by a later stage.

Examples:

-   CSV file
-   dataset
-   run directory
-   metadata file
-   JSON summary

------------------------------------------------------------------------

### Stage Result

Metadata returned after a stage execution.

Typical fields:

-   stage name
-   execution status
-   input artifact path
-   output artifact path
-   processed count
-   execution directory

------------------------------------------------------------------------

## 4. Design Principles

Pipeline orchestration should be handled by a CLI / orchestration layer.

Processing stages should remain independent units of computation.

### Responsibilities of the CLI / Orchestrator

-   Parse pipeline configuration
-   Control stage execution order
-   Pass artifacts between stages
-   Handle failure conditions
-   Generate pipeline summary
-   Persist pipeline execution metadata

### Responsibilities of Processing Stages

-   Perform computation
-   Interpret input artifacts
-   Produce output artifacts

The base processing framework must not manage pipeline dependencies.

------------------------------------------------------------------------

## 5. CLI Behavior

Example CLI invocation:

python -m app.cli.run_batch\
--pipeline stage_a,stage_b\
--config config.yaml

### CLI Option

--pipeline

Defines an ordered list of stages separated by commas.

Example:

stage_a,stage_b

### Mutual Exclusivity

--pipeline and --processor options must be mutually exclusive.

Rules:

-   both specified → error
-   neither specified → error

------------------------------------------------------------------------

## 6. Pipeline Execution Flow

Example pipeline:

Stage A ↓ Stage B

Execution steps:

1.  Execute Stage A
2.  If successful → execute Stage B
3.  If failure → stop pipeline

Pipeline success condition:

All stages must succeed.

------------------------------------------------------------------------

## 7. Artifact Contract

Artifacts produced by one stage become the input for the next stage.

Connection rule:

The orchestrator passes the previous stage result's

output_artifact_path

to the next stage as

input_artifact_path

This avoids re-scanning directories and ensures deterministic execution.

------------------------------------------------------------------------

## 8. Session Resolution

Session identifiers can be derived from runtime context.

Priority order example:

1.  basename(target_dir)
2.  date
3.  default identifier

Session identifiers help organize run artifacts and summaries.

------------------------------------------------------------------------

## 9. Resource Guard (max-images Example)

Pipeline-wide limits may be applied to control resource usage.

Example rule:

-   Resource limit applies to the first stage
-   Later stages operate on the resulting artifacts

Example:

Case 1

limit = 100 stage A output = 100 stage B input = 100

Case 2

limit = 100 actual data = 32 stage A output = 32 stage B input = 32

------------------------------------------------------------------------

## 10. Pipeline Summary

Pipeline execution results should be recorded in a structured format.

Minimum fields:

-   pipeline stages
-   execution date
-   status
-   stage status
-   input/output artifact paths

Example:

{ "pipeline": \["stage_a", "stage_b"\], "date": "2026-02-17", "status":
"success" }

Recommended output file:

pipeline_summary.json

------------------------------------------------------------------------

## 11. Error Handling

Rule:

If a stage fails → do not execute subsequent stages.

Example:

Stage A failure ↓ Stage B skipped

Pipeline success requires:

All stages completed successfully.

------------------------------------------------------------------------

## 12. Compatibility

Single-stage execution must remain supported.

Example:

--processor stage_a

Stages should function both:

-   independently
-   as part of a pipeline

------------------------------------------------------------------------

## 13. Implementation Responsibilities

### CLI / Orchestration Layer

Responsible for:

-   pipeline parsing
-   stage execution order
-   artifact routing
-   summary generation

Example module:

cli/run_batch.py

------------------------------------------------------------------------

### Processing Stage

Responsible for:

-   computational logic
-   artifact generation

Stages should not reference other stages directly.

------------------------------------------------------------------------

### Base Processing Framework

The base framework should not include pipeline logic, such as:

-   stage dependency management
-   artifact routing
-   pipeline orchestration

------------------------------------------------------------------------

## 14. Acceptance Criteria

The system should satisfy the following:

1.  Pipeline execution works via --pipeline
2.  CLI options remain mutually exclusive
3.  Successful stages trigger the next stage
4.  Failed stages stop the pipeline
5.  Artifact passing works correctly
6.  Runtime context propagates correctly
7.  Resource guards function as intended
8.  Pipeline summary is generated
9.  Single-stage execution remains intact

------------------------------------------------------------------------

## 15. Future Extensions

Potential enhancements:

-   pipeline resume capability
-   pipeline execution history
-   multi-stage pipelines
-   generalized artifact abstraction
-   pipeline monitoring and visualization

------------------------------------------------------------------------

## 16. Final Direction

Priority implementation guidelines:

-   Use CLI-based orchestration
-   Keep the base processing framework minimal
-   Standardize artifact contracts between stages
-   Apply resource limits at the pipeline entry stage
