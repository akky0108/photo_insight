# SCRIPT整理設計書（photo_insight）

## 1. 目的
本設計書は、プロジェクト内の shell script（以下 script）を整理し、
過剰な分散・複雑化を防ぎつつ、必要な開発効率化を維持することを目的とする。

---

## 2. 前提
現状の課題：

- script が増えすぎている
- 入口が複数存在し、使用方法が不明確
- Makefile と script の責務が混在している
- GitHub 操作の抽象化により挙動が不透明になりやすい

### コメント
script の削減は目的ではなく、責務を分かりやすくすることが目的である。
そのため、必要な script は残してよいが、役割と配置を明確にする。

---

## 3. 基本方針

### 3.1 原則
- script は最小限に削減する
- 自動化は禁止しないが、用途を限定する
- script は「補助」に限定し、主処理は Python に集約する

### コメント
script を完全禁止にすると開発効率が落ちるため、開発効率化に有効なものは許可する。
ただし、主処理そのものを script に載せるのではなく、補助として使う。

---

### 3.2 役割定義

| 種類 | 役割 |
|------|------|
| Python | 本処理（pipeline, evaluator 等） |
| Makefile | 実行入口・タスク集約 |
| script | 開発効率化・補助処理 |

### コメント
Makefile と script の責務が曖昧になると、修正箇所の判断が難しくなる。
そのため、本処理は Python、入口は Makefile、補助は script に分ける。

---

## 4. GitHub 操作の扱い

### 4.1 方針
GitHub 操作の自動化は開発時に使う機能に限定して許可する。

### 4.2 許可範囲
- Issue 作成
- ブランチ作成補助
- PR 作成補助

### 4.3 制限事項
- 自動マージの禁止
- 状態を隠蔽する処理の禁止
- ブランチ戦略の抽象化禁止

### 4.4 目的
GitHub script は以下のために使用する：

- 入力の簡略化
- コマンドの短縮
- 人為ミスの削減
- ブランチ作成の開発効率化

### コメント
GitHub 操作の自動化は「操作の短縮」を目的とする。
判断を伴う処理（マージ・リリース確定など）は対象外とする。

理由：
過度な自動化は状態の把握を困難にし、トラブル時の復旧を難しくするため。

---

## 5. script 構造設計

### 5.1 原則
- 単体 script の肥大化は禁止
- 構造的分離は許可

### コメント
単体 shell が複雑化すると読解・修正コストが急増する。
一方で、共通処理を切り出した構造化は保守性を高めるため許可する。

---

### 5.2 設計方針

scripts/
├── deploy/
├── dev/
├── github/
│   ├── create_issue.sh
│   ├── create_branch.sh
│   └── create_pr.sh
└── lib/

---

### 5.3 分離ルール

#### 単体 script
- 1責務のみ
- 100 行以内を目安
- 引数と副作用が明確であること

#### 共通処理
- scripts/lib/ に切り出す
- 関数化して再利用する
- 共通化は「重複排除」より「理解しやすさ」を優先する

### コメント
script の肥大化を防ぐため、単体 script は責務を限定する。
複雑な処理は lib に分離し、構造で複雑さを管理する。

理由：
単一ファイルの複雑化は可読性と保守性を著しく低下させるため。

---

### 5.4 NG パターン
- 巨大 script（多機能）
- script → script の多段呼び出し
- ロジックの重複
- 条件分岐の肥大化
- 実行時の暗黙依存

---

## 6. 整理ルール

### 6.1 残す基準
以下を満たす script のみ残す：

- 開発時に繰り返し使用する
- 明確な責務を持つ
- 単体で動作可能
- 内容が理解できる
- 手動実行手順と対応づけられる

### 6.2 削除基準
以下に該当する script は削除：

- 一度しか使っていない
- Makefile の単純ラッパー
- 不透明な挙動
- 過去の一時用途
- 代替手段が明確に存在する

---

## 7. 命名規則

<動詞>_<対象>.sh

例：
- create_branch.sh
- create_pr.sh
- deploy_prod.sh

### コメント
曖昧な名前は用途の誤解を生むため避ける。
名前だけで目的が推測できることを重視する。

### 8.1 必須設定

#!/usr/bin/env bash
set -euo pipefail

---

### 8.2 ログ

echo "[INFO] ..."

---

### 8.3 共通化
- 共通処理は lib に集約
- 再利用可能な形にする
- エラーメッセージは利用者が判断できる文言にする

### コメント
script はトラブル時に読むことが多いため、
正常系よりも異常時の分かりやすさを重視する。

---

## 9. Makefile との関係

### 9.1 原則
- Makefile から script を呼び出してよい
- script は Makefile の補助として機能する

---

### 9.2 役割分離

| 処理 | 実装場所 |
|------|----------|
| 単純処理 | Makefile |
| 開発補助 | script |
| 本処理 | Python |

### コメント
Makefile は入口、script は処理補助として役割を分離する。

理由：
責務を分離することで、変更時の影響範囲を限定し、構造の崩壊を防ぐため。

---

## 10. 移行手順

### 10.1 棚卸し

- 使用頻度確認
- 現在の責務を分類

---

### 10.2 分類
- 残す
- 削除
- lib へ移動
- Python へ移行

---

### 10.3 再構成
- github / dev / deploy に整理
- lib を導入
- Makefile からの呼び出し位置を整理

---

### 10.4 検証
- script 単体での実行確認
- Makefile 経由の実行確認
- README / 運用設計書の更新

---

## 11. 最終原則

GitHub 操作は開発効率化の範囲で使用する。

単体 script はシンプルに保つ。

複雑さは構造で分離する。

script は補助であり、主役ではない。

script は短く、構造は明確に、挙動は可視化する。
