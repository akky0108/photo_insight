# 開発標準フロー設計規約（photo_insight）

## 1. 目的

本規約は、photo_insight プロジェクトにおける開発・運用の標準ルールを定義し、
**個人開発に適したシンプルさ**と、**将来参照可能なドキュメント品質**を両立することを目的とする。

---

## 2. 適用範囲

本規約は以下に適用する。

* 写真の評価・選別機能（①）
* 環境自動化・運用改善（②）
* Git運用
* CI/CD
* デプロイ

---

## 3. 基本原則

### 3.1 開発フロー

すべての変更は以下の順序で実施する。

1. 設計（簡素）
2. 実装
3. テスト
4. 設計書更新（恒久）
5. develop へ反映
6. main へ反映
7. 本番デプロイ

---

### 3.2 分離原則

以下は必ず分離する。

* 機能開発（①）と環境自動化（②）
* PR単位の責務
* 実装と運用ロジック

---

### 3.3 PR原則

* 1PR = 1テーマ
* 小さく分割して develop に反映
* 機能変更と環境変更を混在させない

---

## 4. ブランチ戦略

### 4.1 ブランチ構成

* develop: 開発統合
* main: リリース

### 4.2 命名規則

#### 機能系

* feat/<name>
* fix/<name>

#### 環境・改善系

* chore/<name>
* refactor/<name>

---

## 5. 設計書運用

### 5.1 基本方針

設計書は以下の2種類とする。

| 種類   | 方針        |
| ---- | --------- |
| 作業設計 | 簡素・短時間で作成 |
| 恒久設計 | 将来参照可能な品質 |

---

### 5.2 作業設計（簡素）

作業前に最低限以下のみ記述する。

* 目的
* 変更点（何をどう変えるか）
* 完了条件

※ 5分以内で書ける粒度とする

例：

```id="cxf3xj"
目的: portrait_quality の閾値調整
変更点: blur_threshold を 0.4 → 0.5
完了条件: ranking結果が期待通りになる
```

---

### 5.3 恒久設計（重要）

実装完了後に更新する設計書は以下を満たすこと。

* なぜその設計になっているかが分かる
* 入出力が明確
* 処理の流れが追える
* 将来読んでも理解できる

最低限の構成：

* 概要
* 入力
* 出力
* 処理内容
* 注意点（制約・前提）

---

## 6. 自動化方針

### 6.1 基本方針

自動化は禁止しない。
ただし以下を必須とする。

* 処理内容が明示的
* 手動で再現可能
* デバッグ可能

---

### 6.2 自動化レベル

| レベル | 内容       | 方針 |
| --- | -------- | -- |
| Lv1 | 単一処理     | 推奨 |
| Lv2 | 手順集約     | 許可 |
| Lv3 | 一括処理     | 注意 |
| Lv4 | ブラックボックス | 禁止 |

---

### 6.3 GitHub操作

自動化は許可する。

#### 許可

* Issue作成
* PR作成
* Release補助

#### 非推奨

* 自動マージ
* 状態の隠蔽

---

## 7. Makefile設計

### 7.1 基本方針

Makefileの複雑化は許可する。
ただし構造化を必須とする。

---

### 7.2 設計ルール

* 用途別に分類する
* 1ターゲット1責務
* ネストや呼び出しは明示的にする

---

### 7.3 推奨構造

```make id="nd3r1r"
# 開発
fmt:
lint:
test:
test-light:

# 実行
run:
run-full:

# CI
ci:

# デプロイ
deploy:

# GitHub
gh-pr:

# ユーティリティ
clean:
```

---


---

```bash id="bm93zy"
make fmt
make lint
make test-light
```

---

## 9. デプロイ

### 9.1 条件

* テスト完了
* 設計書更新済み
* develop で安定

---

### 9.2 実行

* Docker Compose ベース
* ログ確認
* 出力確認

---

## 10. 運用ルール

### 10.1 手動と自動の使い分け

| 種類     | 方針  |
| ------ | --- |
| 繰り返し処理 | 自動化 |
| 判断が必要  | 手動  |
| デプロイ   | 半自動 |

---

### 10.2 禁止事項

* ブラックボックス化
* 意図不明な処理
* 不可逆な自動化

---

### 10.3 推奨事項

* 小さく変更する
* 明示的に実行する
* ログを残す
* 再現性を維持する

---

## 11. フェーズ戦略

### フェーズ1（安定化）

* シンプル運用確立

### フェーズ2（改善）

* 構造整理

### フェーズ3（再設計）

* アーキテクチャ見直し

---

## 12. 最終原則

> 設計は簡素に、記録は将来読める形で残す。

> 自動化は許可するが、必ず制御可能であること。

> シンプルさと再現性を最優先とする。
